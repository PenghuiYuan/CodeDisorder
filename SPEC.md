# CodeConfuse.Web —— 代码混淆 Web 服务需求文档

> 目标:为 C / C++ / Python / Java / Go 源码提供 **AST 级别** 的结构混淆,生成
> 语义等价、可在目标 OJ(Codeforces / 牛客 / AtCoder / 学习通 / PTA)直接提交且不被
> 查重系统判定为重复的完整可编译源码。
> 服务面向**任意访客**,无登录、无付费、无长期存储。
>
> **后续会考虑上线 + 登录 + 付费**,但**当前 MVP 不实现**,详见 §15「上线 / 付费 / 登录 —— 扩展点预留」。

## 1. 背景与目标

### 1.1 背景

`CodeConfuseTool` 是一款本地 Qt 桌面工具,基于文本级解析对 C / C++ / OC 源码做
标识符改名 + 垃圾代码注入 + `#define` 宏方案输出。在结构相似度查重日益普及
(教学型 OJ 用 Moss / SimHash,部分商业 OJ 用 AST 相似度)的背景下,仅做"改名"
已不够。

### 1.2 目标

- 提供一个 **Web 服务**,接收用户粘贴的源码,返回 **可直接提交到目标 OJ** 的混淆后源码。
- 混淆在 **抽象语法树(AST)级别** 完成,改名 / 控制流改写 / 表达式拆分 / 垃圾代码插入等
  多策略组合,使结构相似度低于查重阈值。
- 对 **5 种 OJ**(Codeforces / 牛客 / AtCoder / 学习通 / PTA)分别提供不同强度的预设,
  适配各自查重策略。
- 全程 **不存储用户代码**,无账号,刷新即丢。

### 1.3 非目标

- 不在 OJ 上真正"提交并验证运行结果"(只做本地编译/语法验证,确保语法和类型层面没破)。
- 不支持"源语言 → 另一种目标语言"的转写(如 C → Python)。
  > 注:虽然 UI 允许源/目标语言不同,但混淆后的目标语言**必须仍是源码所属语言**(见 §14 解释)——本服务不做跨语言翻译。
- 不做"绕过人工审核 / 绕过 OJ 账号风控"的事情。
- 不保证对**所有 OJ** 在所有时间点都有效;查重算法会持续升级,本服务只能针对已知
  公开算法提高抵抗能力。
- **MVP 阶段不实现用户体系**;登录注册、付费、配额等用户态能力留到上线版本(详见 §15)。

## 2. 用户与场景

### 2.1 目标用户

**通用 Coder(任意访客)**:打开网页 → 粘贴代码 → 选目标 OJ → 拿混淆结果 → 复制走。
不注册、不登录、不付费、不留存历史。

### 2.2 核心场景

1. **比赛/作业前快速过查重**:用户在 OJ 上交作业前,把已 AC 的代码贴进页面,
   得到一份结构不同的等价代码,再贴回 OJ 提交。
2. **批量生成多个变体**:同一份代码想要 3 / 5 / 10 份不同混淆结果(便于挑选最合适的一份提交),用 .zip 一次性下载。
3. **测试 / 调试混淆器本身**:用不同 OJ 的预设进行 A/B 对比,看哪种输出更"安全"。

### 2.3 禁止用途(显式声明)

- 学术不端 / 作业抄袭 / 比赛代打。
- 商业代码保护(本服务对上传代码无保密能力,且会变换结构)。
- 任何违反目标 OJ 服务条款的用途。
- **用户协议 / 隐私政策 / 退订机制**待上线版本提供,详见 §15。

页面底部放"使用须知"链接,首次访问弹出"勿用于学术不端"提示。

## 3. 范围与功能

### 3.1 支持的源语言

| 语言 | 编译器/工具链 | 解析器 | 状态 |
|---|---|---|---|
| C | gcc / clang | libclang | MVP 必须 |
| C++ | gcc / clang | libclang | MVP 必须 |
| Python | CPython 3.x | `ast` 模块 | MVP 必须 |
| Java | OpenJDK 17+ | JavaParser / Eclipse JDT | MVP 必须 |
| Go | go 1.21+ | `go/ast` | MVP 必须 |

> MVP 之外的扩展语言:留接口但不实现(后续 Kotlin / Rust / C# / JavaScript 等可后插)。

### 3.2 支持的目标 OJ 与预设

每个 OJ 预设是 **一组混淆策略的开关集合**,而非"一种算法"。

| OJ | 预设名 | 强度 | 策略组合 |
|---|---|---|---|
| Codeforces | `cf-strict` | 弱 | 标识符改名 + 简单控制流平坦化 |
| AtCoder | `atcoder-strict` | 弱 | 同上,无垃圾代码 |
| 牛客 | `nowcoder-mid` | 中 | 改名 + 控制流改写 + 局部表达式拆分 |
| PTA | `pta-strong` | 强 | 改名 + 控制流改写 + 表达式拆分 + 垃圾代码 + 函数拆分 |
| 学习通 | `xuexitong-strong` | 强 | 同 PTA,且模板参数随机化 |

UI 上提供"自定义"档,逐项勾选策略,作为高级用户选项。

### 3.3 混淆策略集合(实现维度)

| 编号 | 策略 | 适用语言 | 作用 |
|---|---|---|---|
| S1 | 标识符改名(变量/函数/类/参数) | 全部 | 改变 token 序列,降低文本相似度 |
| S2 | 字面量改写(数字基底/字符串切分) | C / C++ / Java / Go | 改变 token,但要保留类型 |
| S3 | 控制流平坦化(state machine 化) | C / C++ / Java / Go | 改变 AST 形状,破坏结构相似度 |
| S4 | 死代码 / 不透明谓词插入 | C / C++ / Java / Go | 改变 CFG,但保持语义 |
| S5 | 表达式拆分 / 变量重命名中间值 | 全部 | 破坏表达式子树相似度 |
| S6 | 函数拆分(长函数切成多个调用) | C / C++ / Java / Go | 降低函数级相似度 |
| S7 | 模板参数 / 泛型参数随机化 | C++ / Java / Go | 改变类型实例化路径 |
| S8 | 注释剥离 + 空白归一化 | 全部 | 与改名无关的通用前处理 |
| S9 | include / import 顺序随机 | 全部 | 微调,几乎只影响文本指纹 |

### 3.4 MVP 必做 / 后续可选

- **MVP 必做**:S1、S2(限数字字面量)、S3(单层)、S4(简单不透明谓词)、S5(限二元表达式)、S8、S9。
- **后续**:S6、S7、按 OJ 预设差异化细节调参。

### 3.5 验证能力

- 服务端对混淆结果 **做一次本地编译/语法检查**
  (C/C++ 用 gcc/clang 编译到目标、不链接;Python 用 `ast.parse`;Java 用 `javac -proc:none`;Go 用 `go vet` 之类),
  失败则在响应里给出**可读错误**而不是垃圾。
- **不在 OJ 上提交**,也不在沙箱里跑样例。

## 4. 总体架构

### 4.1 系统视图

```
┌────────────┐    POST /api/confuse    ┌───────────────┐
│  Web 前端  │ ──────────────────────▶ │  API 网关/WSGI│
└────────────┘                          └──────┬────────┘
                                                │
                                                ▼
                                     ┌──────────────────────┐
                                     │  任务调度(短任务同步, │
                                     │  重任务入异步队列)     │
                                     └──────┬───────────────┘
                                            │
                       ┌────────────────────┼─────────────────────┐
                       ▼                    ▼                     ▼
                ┌─────────────┐      ┌─────────────┐       ┌─────────────┐
                │ Worker: C/C++│     │ Worker: Java │       │ Worker: Go   │
                │ (libclang)  │      │ (JavaParser) │       │ (go/ast)     │
                └──────┬──────┘      └──────┬───────┘       └──────┬───────┘
                       │                    │                       │
                       ▼                    ▼                       ▼
                ┌─────────────┐      ┌─────────────┐       ┌─────────────┐
                │ gcc/clang 验证│    │ javac 验证  │       │ go vet 验证  │
                └─────────────┘      └─────────────┘       └─────────────┘
```

> §15 上线时,会在「API 网关」与「任务调度」之间加一层"用户态 + 配额"中间件(详见 §15.3)。

### 4.2 模块划分

- **前端**:单页应用,代码编辑器(Monaco Editor)+ 选项区 + 结果区。开源前端,无后端状态。
- **API 层**:无状态 HTTP 接口,接收混淆请求、调度 Worker、返回结果;**不持久化任何东西**。
- **Worker 层**:每种语言一个独立进程/容器,接收源码字符串 + 策略开关,产出混淆后源码字符串。
- **验证层**:Worker 内部子模块,负责本地编译/语法验证。
- **资源中心**:维护公共的"标识符词表"(如英文词、混淆用的无意义符号名)及"不透明谓词库"。

### 4.3 部署拓扑

- 单体 Web 服务 + 内部多 Worker;容器化部署,可水平扩缩。
- 不引入数据库;**用户数据只在请求生命周期内存在**。
- 日志与指标走标准收集,不留存代码内容,只留元信息(语言、长度、用时、状态码、错误类型)。

## 5. 关键流程

### 5.1 用户流程(端到端)

1. 打开首页 → 选择源语言、目标语言、目标 OJ(默认 PTA-strong)→ 选择"生���数量"(默认 1)→ 粘贴代码 → 点击"开始混淆"。
2. 前端 POST `/api/confuse`,载荷:语言、目标 OJ、生成数量、策略覆写(可选)、源码。
3. API 路由到对应 Worker,Worker 解析 → 应用策略 → 验证 → 返回结果(单文件返回字符串;多文件返回 .zip 的 base64)。
4. 前端按 `count` 分支展示:
   - `count = 1`:右栏直接显示混淆后代码,提供"复制 / 下载单文件"按钮。
   - `count > 1`:右栏只显示第 1 个变体 + "已生成 N 个,共 X KB,点此下载 .zip"按钮。
5. 用户复制到 OJ 提交;**不与服务端有进一步交互**。

### 5.2 混淆流程(Worker 内部)

```
源码字符串
  ↓
[预清洗]   去除 BOM、统一换行、剥离注释
  ↓
[解析]     抽象语法树(AST)
  ↓
[策略选择] 按 OJ 预设 + 用户覆写,确定本轮启用的策略
  ↓
[变换]     按 AST 节点遍历应用策略(S1..S9)
  ↓
[代码生成] AST → 目标语言源代码
  ↓
[验证]     编译/语法检查;失败则尝试弱化策略重跑 1~2 次,仍失败则返回错误
  ↓
输出字符串
```

### 5.3 失败处理

- **解析失败**:源码本身有语法错,直接返回解析错误(带行号 / 列号),不做部分混淆。
- **策略冲突**:用户同时勾选了互斥选项,前端做防抖,后端做兜底校验。
- **验证失败**:先按"先去掉垃圾代码 → 再去掉控制流改写 → 仅保留改名"做降级,
  降级后若还失败则返回错误与原始源码。
- **超时**:单次混淆超过 10s 直接取消,降级方案再跑 5s。
- **批量场景部分失败**:`count > 1` 时某几个变体失败,zip 里只放成功的;响应里 `failed_indexes` 标注哪些编号失败。

## 6. 接口设计

### 6.1 主接口:POST `/api/confuse`

请求:

```json
{
  "language_in":  "cpp",                  // c / cpp / python / java / go
  "language_out": "cpp",                  // c / cpp / python / java / go,默认与 language_in 相同
  "preset":       "pta-strong",           // 见 3.2
  "count":        1,                      // 1 / 3 / 5 / 10,默认 1
  "overrides": {                          // 可选,逐项覆盖
    "rename": true,
    "flatten": true,
    "junk": true,
    "splitFunction": false
  },
  "code": "string, <= 200KB, UTF-8"
}
```

> 注:`language_in` 与 `language_out` 是两个独立字段,UI 上对应两个下拉。MVP 阶段 `language_out` 必须与 `language_in` 相同(本服务不做跨语言翻译),后端校验:不相等时返回 400。

响应 —— **`count = 1`**:

```json
{
  "status": "ok",
  "language_in": "cpp",
  "language_out": "cpp",
  "preset": "pta-strong",
  "count": 1,
  "applied": ["rename", "flatten", "junk"],
  "code": "string, UTF-8",
  "verify": "compiled"                  // compiled / syntax-ok / warning
}
```

响应 —— **`count > 1`**:

```json
{
  "status": "ok",
  "language_in": "cpp",
  "language_out": "cpp",
  "preset": "pta-strong",
  "count": 5,
  "applied": ["rename", "flatten", "junk"],
  "zip_b64": "base64 string, 文件名 confused_<n>.zip, 内含 N 个文件:confused_001.<ext> .. confused_NNN.<ext>",
  "verify": "compiled",                 // 任一文件编译失败时退化为 "warning"
  "failed_indexes": []                  // 失败变体编号列表(从 1 开始);全成功时为空数组
}
```

响应 —— **失败**:

```json
{
  "status": "error",
  "code": "parse_error | verify_error | quota_exceeded | auth_required | ...",  // 业务错误码,见 §15.6
  "stage": "parse | transform | verify",
  "message": "string, 人类可读",
  "errors": [
    {"line": 12, "column": 4, "message": "..."}
  ]
}
```

### 6.2 辅助接口

- `GET /api/presets`:返回所有 OJ 预设的元信息(名称、强度、包含的策略)。
- `GET /api/strategies`:返回所有可用策略的元信息(编号、说明、适用语言、是否互斥)。
- `GET /api/health`:健康检查,返回 200。
- `GET /`:单页应用入口。
- `GET /tos`:服务条款(纯静态页)。
- **§15 上线时追加**:`/api/auth/signup`、`/api/auth/login`、`/api/auth/logout`、`/api/auth/me`、`/api/billing/*`、`/api/quota` 等。

### 6.3 限制

- 单请求代码大小 ≤ 200KB(超出 413)。
- 单 IP 并发 2(超出 429)。
- 不接受压缩(便于审计与日志)。
- 不持久化输入输出。
- `count` 取值限定为 1 / 3 / 5 / 10(后端校验,其他值 400)。
- **§15 上线后追加**:
  - 匿名用户:每天 N 次(默认 5,具体值 §15.5 敲定) → 超出 429 `code=quota_exceeded`。
  - 登录用户:按订阅档位(免费 / 会员)。

## 7. 数据与资源

### 7.1 输入数据

- 用户上传的源码,仅在请求生命周期存在。
- **不写盘**(Worker 用内存流处理,验证阶段用临时文件,处理完即删)。
- 日志只记"语言、长度、用时、状态、错误类型",不记代码内容。

### 7.2 资源数据(只读,部署时打包)

| 资源 | 形式 | 大小预估 | 用途 |
|---|---|---|---|
| `wordlist.txt` | 文本 | ~3 MB | 标识符改名用词表 |
| `opaque_predicates.json` | JSON | ~50 KB | 不透明谓词模板 |
| `language_settings/*.json` | JSON | ~10 KB / 语言 | 各语言 AST 节点映射、保留字表 |
| `templates/<lang>/*` | 文本 | ~50 KB / 语言 | 兜底代码模板 |

### 7.3 内部数据流

请求 → API 内存 → Worker 内存 → 输出。
**无数据库、无文件系统读写**(验证阶段除外,且临时文件用 `mkstemp` 即用即删)。

> §15 上线时,会新增 **用户 / 订阅 / 用量计数** 三张表(详见 §15.3),仍不存代码内容。

## 8. 需求:非功能性

### 8.1 性能

- 50KB 以下 C++ 源码 P95 延迟 < 3s(单 Worker 节点,`count = 1`)。
- 100~200KB 源码 P95 延迟 < 8s。
- `count = 10` 时 P95 延迟 < 25s(线性扩展预期)。
- 验证阶段不计入主延迟(若超过 2s,改异步返回;MVP 阶段同步返回,长文件降级)。

### 8.2 可用性

- 99% 月度可用性(无 SLA 承诺,仅参考值)。
- 单 Worker 节点故障不影响 API 整体接收(失败请求立即返回 503,不排队)。

### 8.3 安全

- **不存代码**,意味着被入侵后能拿到的也只是日志里的元信息。
- 限制上传大小与并发,做基本 DoS 防护。
- 防止"上传恶意代码 → 服务端 gcc/llvm 漏洞利用":Worker 进程以最低权限运行,无网络出站,只能写 `/tmp`。
- 前端 CSP 严格,禁外链脚本。

### 8.4 隐私

- 不存代码 → 规避 GDPR / 个保法对"代码即个人数据"的争议。
- 服务条款明示"代码不持久化",并在请求响应里加 `Cache-Control: no-store`。

### 8.5 兼容性

- 现代浏览器(Chrome / Edge / Safari / Firefox 最近 2 个大版本)。
- 移动端可读、可操作,但不强求完美布局。

### 8.6 可观测

- 暴露 `/metrics`(Prometheus 格式):QPS、P95 延迟、失败率、各语言分布、各预设分布、`count` 分布。
- 日志结构化(JSON),含 `trace_id`;不含代码内容。
- **§15 上线后追加**:`user_id`(已登录时)、`subscription_tier`、匿名/已登录分布等指标。

## 9. 安全与合规要点

- **不存代码**:从架构上根除"被入侵导致用户代码泄露"。
- **服务条款 + 提示**:页面底部固定"勿用于学术不端"声明,首次访问弹窗提示。
- **沙箱**:Worker 进程以低权限用户运行,禁外网,禁文件系统写(除 `/tmp` 临时文件)。
- **资源限制**:每个 Worker 进程 cgroup 限制 CPU / 内存,防止恶意大文件 / 死循环。
- **CSP**:前端禁 inline script、外链脚本。
- **速率限制**:基于 IP + UA 指纹,超限返 429。
- **审计日志**:记录"被拒绝的请求"(超大文件、频繁 429),便于追溯滥用。
- **法律边界**:本服务不绕过 OJ 自身的服务条款,不做账号代登、不做提交流水伪造。

> §15 上线时,需补充"用户协议 / 隐私政策 / 退订 / 注销账户 / 发票"等合规条款,详见 §15.7。

## 10. 风险与对策

| 风险 | 等级 | 对策 |
|---|---|---|
| 查重算法升级导致混淆失效 | 中 | 预设可独立迭代,后台策略可热更新 |
| libclang / JavaParser 版本差异导致 AST 不同 | 中 | 固定编译器版本,Docker 镜像锁版本 |
| 恶意大文件 / 死循环 DoS | 中 | 大小限制 + Worker cgroup 限制 + 进程超时 |
| 用户代码被误判"含有恶意内容" | 低 | 不存代码,无审核环节,本服务侧无此风险 |
| 学术不端争议 | 中 | 显著位置提示,服务条款约束;技术上不再做识别/拦截 |
| 解析器在某些语法上崩溃 | 中 | Worker 异常隔离,崩了记日志,不影响其他请求 |
| `count > 1` 时 zip 体积过大被截断 | 低 | 单 zip 限制 5MB,超出时分卷 zip 或后端流式响应 |
| §15 上线后:被刷免费配额 | 中 | 验证码、IP + UA 指纹、用户态配额计数 |
| §15 上线后:用户协议 / 退订 / 注销 不合规 | 中 | 上线前法务审核;预留注销 / 退款状态机 |

## 12. 验收标准(M1)

- [ ] 用户粘贴 50KB 以内的 C++ 代码,3s 内拿到可编译输出。
- [ ] 至少通过编译验证(`g++ -std=c++17 -fsyntax-only`)。
- [ ] 输出与输入在语义上等价(对固定输入样例,运行结果一致)。
- [ ] 输出不与输入文本相似度高于阈值(Jaccard < 0.5,AST 编辑距离 > 阈值,
      具体数字在 M1 末敲定)。
- [ ] 服务端日志不含代码内容。
- [ ] 触发解析错误时返回 200 + `status: error`,并给出错误位置。
- [ ] 压测:单 Worker 节点 10 并发,无崩溃,P95 < 5s。
- [ ] **MVP 阶段不引入任何形式的用户态**:无登录、无付费、无配额(无限次数)。这条作为 MVP 的硬性验收项。

## 13. 后续待定事项(进入设计阶段前需拍板)

> 以下条目本次未与你确认,文档给出默认方案,可在评审时一并敲定。

1. **部署形态**:单体 Web 服务 + 队列式后端混淆 Worker。
2. **混淆强度分级**:暴露 5 档 OJ 预设 + 自定义开关。
3. **代码是否短暂缓存**:不缓存,纯无状态。
4. **商业化模式**:个人项目,免费、无广告、无登录(MVP);上线时再上付费(详见 §15)。
5. **后端是否做编译验证**:是(只跑编译/语法,不在 OJ 上跑样例)。
6. **团队/协作能力**:不支持。
7. **部署平台**:推荐容器云,可水平扩缩。
8. **隐私与服务条款**:页面底部固定提示"勿用于学术不端";上线后补完整用户协议(§15.7)。
9. **历史/版本管理**:不支持,刷新即丢。
10. **国际化**:暂只中文。
11. **性能指标**:单文件 < 200KB、混淆 < 3s、P95 延迟 < 5s。
12. **API 是否对外开放**:仅 Web 前端,不开放 API(指 MVP;上线时考虑开放付费 API,见 §15.8)。
13. **多文件输入**:MVP 只支持单文件,后续 M4 再做 zip 上传。
14. **开源与 license**:Web 前端 + 后端均开源,license 待选(Apache 2.0 / MIT / AGPL)。
15. **错误信息透明度**:解析错误返回行号、列号、token(帮助用户改源码);
    也可能让攻击者更易构造恶意输入,后续视情况做脱敏。
16. **保留字表来源**:每种语言手工维护一份系统/标准库保留字表,启动时加载。
17. **沙箱深度**:Worker 进程用 cgroup 限制;seccomp / Landlock 留作 M3+。
18. **降级策略**:验证失败时按"去垃圾 → 去控制流改写 → 仅改名"三档降级,UI 显式提示"已降级"。
19. **代码长度上限**:MVP 200KB,OJ 场景通常 < 64KB,可考虑砍到 100KB。
20. **与 CodeConfuseTool 桌面版的关系**:致谢并链接;暂不强制品牌统一。
21. **生成数量上限**:当前定 1 / 3 / 5 / 10,是否需要 20 / 50 的大批量档?
22. **zip 内文件命名**:`confused_001.<ext>` 顺序编号;是否需要保留原文件名信息?

> **以下条目属于 §15 上线阶段,MVP 不需要敲定,但进入 M5 前必须敲定**:
23. **计费维度**:按"次"还是按"字符数"计费(影响前端展示与后端计数埋点)。
24. **支付厂商**:国内(微信 / 支付宝)或国外(Stripe),或两者都要。
25. **免费档配额**:匿名 / 已注册 / 已付费 三档,每天 N 次的具体值。
26. **会员档权益**:无限次 / 更高批量上限(`count > 10`)/ 高级预设 / API 开放等。

## 14. 翻译器式 UI 规范(Google 翻译风格)

### 14.1 设计目标

- 用户在 **3 秒内** 理解整个界面的用法(不需要说明文档)。
- 输入 / 输出严格对称,左右两栏的操作完全一致(粘贴、清空、复制)。
- "源语言 → 目标语言"与"OJ 预设"两个维度**正交**——选 OJ 不影响语言选择,选语言不影响 OJ。

### 14.3 控件规格

#### 14.3.1 顶栏

- 左侧:网站标题(`CodeConfuse.Web`),点击回首页。
- 右侧:「使用须知」「服务条款」「GitHub」三个文字链接。
- **§15 上线时**:再追加「登录」/「注册」按钮(MVP 不显示)。
- 整体高度 ≤ 48px。

#### 14.3.2 源语言下拉(左下角)

- 选项:`C` / `C++` / `Python` / `Java` / `Go`,与 §3.1 一致。
- 右侧实时显示"字符数 N · 行数 N"(每 200ms 节流更新)。
- 上限 200KB,超出时:
  - 编辑器边框变红;
  - 「开始混淆」按钮禁用并 hover 提示"代码超过 200KB 上限";
  - 字符数变红。

#### 14.3.3 目标语言下拉(右下角)

- 选项与左下角相同,**默认与源语言同步**。
- 当用户手动改为与源不同:弹一次小提示"本服务不支持跨语言翻译,目标语言将自动改回与源相同",然后下拉自动回滚。
  > 实现细节:目标语言是后端校验硬约束,前端做"友好提示 + 防止误操作"。
- 字符数显示的是"当前已生成的混淆结果"的字符数,运行前显示 `—`,运行失败时显示 `×`。

#### 14.3.4 源编辑器(左栏)

- 库:Monaco Editor,与 VS Code 同款,免费、可商用。
- 语言着色根据"源语言下拉"实时切换。
- 支持 Tab 缩进、自动闭合括号 / 引号。
- 顶部留 1 行工具条:[清空] [粘贴] (粘贴调 `navigator.clipboard.readText()`)。
- 编辑器内容变化时,右栏保持**不变**(右栏是"上一次运行的结果")。

#### 14.3.5 结果区(右栏)

- 同样使用 Monaco Editor,但 `readOnly: true`,语言着色根据"目标语言"。
- 三种状态:
  1. **空闲**(未运行 / 已清空):显示提示文字"等待混淆结果..."。
  2. **运行中**:显示"正在生成第 N / M 个变体..."的进度,编辑器内显示当前已完成的最后一个。
  3. **完成**:
     - `count = 1`:显示完整结果,可用 [复制] [下载单文件] 两个按钮。
     - `count > 1`:显示第 1 个结果 + 顶部条带"已生成 N 个,共 X KB",只显示 [下载 .zip] 按钮,**[复制] [下载单文件] 隐藏**。
  4. **失败**:显示错误信息(行号、列号、错误描述),隐藏所有结果按钮,显示"修改源码"按钮(点击聚焦到左栏对应行)。

#### 14.3.6 中栏控制

- **OJ 预设下拉**:5 个 OJ + 1 个"自定义"。"自定义"选中时,展开右侧"自定义策略"折叠区(S1~S9 共 9 个开关)。
- **生成数量下拉**:选项 `1` / `3` / `5` / `10`,默认 `1`。
  - 选择 > 1 时,旁边出现文字提示:"批量生成较慢,只支持 .zip 下载"。
- **自定义策略折叠区**(默认折叠):S1~S9 共 9 个开关 + 每个策略一行说明,展开后高度 ≤ 280px。
- **开始混淆按钮**:主按钮,蓝色填充,宽度 240px,高度 48px。
  - 禁用条件:左栏为空 / 字符数 > 200KB / 有运行中的请求。
  - 点击后:按钮变为进度条 + 状态文字(2s 内完成的简单混淆显示"已完成",复杂场景显示"已应用 S1, S3, S4...")。

#### 14.3.7 底栏

- 固定文字:`勿用于学术不端 · 不存储代码 · 不做账号登录`。
- **§15 上线时**:追加 `用户协议` / `隐私政策` 链接;`不做账号登录` 改为"已登录时显示当前用户"。
- 高度 32px,字号 12px,灰色。

### 14.4 输入输出规则

- **左栏 ↔ 右栏内容互不联动**:左栏修改,右栏保持上一次结果;右栏复制 / 下载不影响左栏。
- **字符数同步**:仅左栏在用户输入时更新字符数;右栏字符数在"完成"时更新,反映"本次结果"的字符数。
- **超限行为**:`code.length > 200 * 1024` 时:
  - 前端直接拦截,不发请求,显示提示。
  - 服务端兜底校验同样 413。

### 14.5 数量选择与批量行为

| count | 右栏显示 | 可用按钮 | 响应字段 |
|---|---|---|---|
| 1 | 完整结果 | 复制、下载单文件 | `code`(单字符串) |
| > 1 | 第 1 个 + "已生成 N 个,共 X KB" | 下载 .zip(其他按钮隐藏) | `zip_b64`、`failed_indexes` |

- `.zip` 内文件命名:`confused_001.<ext>` … `confused_0NN.<ext>`,`<ext>` 按目标语言决定(`cpp` / `c` / `py` / `java` / `go`)。
- `.zip` 大小硬上限 5MB(超过则后端 400,前端提示"批量结果过大,请减少数量")。
- `count` 仅允许 1 / 3 / 5 / 10(后端 400 兜底,前端禁用其他值)。
- 部分成功(比如 5 个里 2 个失败):`.zip` 里只放成功的,响应里 `failed_indexes: [3, 5]`,UI 在下载按钮旁显示"5 个中 3 个成功,2 个失败"。

### 14.6 状态与错误展示

| 状态 | 触发条件 | UI 表现 |
|---|---|---|
| 空闲 | 首次打开 / 清空后 | 右栏提示文字 |
| 解析失败 | 用户代码有语法错误 | 右栏错误视图,行号 / 列号 / 错误信息;左栏对应行高亮 |
| 验证失败(已降级) | 编译报错,降级后通过 | 右栏结果顶部红色条"⚠ 已降级:去掉了 S4 死代码插入" |
| 验证失败(无法挽救) | 降级后仍失败 | 右栏错误视图 + "复制原始代码以便排查"按钮 |
| 部分成功(count > 1) | 批量时部分失败 | 见 §14.5 |
| 全失败 | 任何阶段彻底失败 | 右栏错误视图,无下载按钮 |
| 超时 | 服务端 10s 未返回 | 前端超时弹窗,允许"重试" |
| §15:超出配额 | 登录用户订阅档位用尽 / 匿名用户日上限 | 右栏错误视图,`code=quota_exceeded`,提供"升级"或"登录"按钮 |
| §15:未登录使用付费功能 | 已订阅但 token 过期 | 右栏错误视图,`code=auth_required`,提供"重新登录"按钮 |

### 14.7 响应式

- **桌面端(≥ 1024px)**:两栏并排,左右各占 50%。
- **平板端(768~1023px)**:两栏并排,各 50%,字号缩 0.9x。
- **移动端(< 768px)**:两栏**上下堆叠**,左栏在上,右栏在下;中栏控制条固定在两栏之间;顶栏 + 底栏常驻。
- 所有断点下,「开始混淆」按钮始终可见(移动端粘性置底)。

### 14.8 显式不做(MVP 范围外)

- **多文件输入**:用户一次粘贴多文件(.h + .cpp 一起),MVP 直接按第一个文件处理,不合并、不提示(留作 M4)。
- **跨语言翻译**:`language_in != language_out` 视为非法,后端 400。
- **撤销 / 重做**:不做。运行后改左栏内容,右栏不变;用户想换结果就再点「开始混淆」。
- **本地保存历史**:不做,刷新即丢。如需要可让用户用浏览器自带的 IndexedDB(后续 M4)。
- **账号 / 收藏 / 分享链接**:不做,违反 C5(指 MVP;上线后由 §15 提供)。
- **拖拽上传文件**:MVP 不做,仅支持粘贴;后续可加。
- **目标语言与源不同时的代码生成**:不做(见 §1.3)。

### 14.9 与 §6 接口的对应

- §6.1 的 `count` 字段是 UI 上"生成数量"下拉的直接映射。
- §6.1 响应里的 `code` 字段对应 `count = 1` 的右栏显示。
- §6.1 响应里的 `zip_b64` 字段对应 `count > 1` 的"下载 .zip"按钮。
- §6.1 响应里的 `language_in` / `language_out` 是两个下拉的当前值(给后端日志用)。
- §6.3 的"count 仅允许 1/3/5/10"与 §14.5 的"批量行为"是同一约束的两端。

## 15. 上线 / 付费 / 登录 —— 扩展点预留(★ MVP 不实现,M5 启动前敲定 ★)

> 本节是**待办清单**,不是 MVP 任务。MVP 阶段所有实现者可以**完全忽略**本节,只需保证 §15.4 列出的"扩展点预留"被遵守。

### 15.1 触发条件(什么时候进入 M5)

必须**全部满足**:

1. **MVP 全部验收标准(§12)通过**。
2. §13.23~26 列出的"上线相关"待定项已敲定。
3. 法务 / 财务 / 运营流程对接完成(用户协议审核、支付通道开通、发票资质)。
4. §15.4 描述的"MVP 阶段扩展点预留"已被遵守,改造工作量可控。

### 15.2 上线后用户态总览

- **匿名用户**:可不注册使用,每日 N 次(默认 5,§15.5 敲定);超出后提示登录或注册。
- **注册用户**:邮箱 + 密码(或第三方登录),免费档每天 N 次。
- **付费用户(会员)**:订阅期内无限次,或按"次 / 字符数"包月。
- **管理员**(内部使用,非用户功能):运营看板、看用量、强制下线。

### 15.3 架构改造点(后端)

按"插拔式中间件"思路设计,MVP 阶段不实现,但接口路径要预留:

| 位置 | 现状(MVP) | 上线后 |
|---|---|---|
| 入口 | `/api/confuse` 直接处理 | 入口前加"用户态 + 配额"中间件,根据 `Authorization` 头判定匿名 / 登录 / 会员 |
| 错误响应 | `{status, stage, message, errors}` | 追加 `code` 字段(`quota_exceeded` / `auth_required` / `subscription_expired` / ...),见 §15.6 |
| 数据存储 | 无 | 新增 Postgres / SQLite,只存:用户表、订阅表、用量计数表(仍不存代码) |
| 鉴权 | 无 | session(cookie)或 JWT(token),上线前二选一 |
| 支付 | 无 | 第三方支付接入,异步回调;状态机:待支付 / 已支付 / 已退款 / 已过期 |
| 审计 | 仅请求级日志 | 追加"用户级用量日志"(按 `user_id` 聚合),便于客服查询 |

### 15.4 MVP 阶段的"不破坏"约束(★ 实现者必读 ★)

为了让 §15.3 的改造可控,MVP 阶段必须做到:

1. **API 路径稳定**:只追加 `/api/auth/*`、`/api/billing/*`、`/api/quota` 等路由,不动 `/api/confuse` 的 URL、HTTP 方法、请求/响应字段。
2. **错误响应结构稳定**:MVP 失败的 JSON 已是 `{status, code, stage, message, errors}`(`code` 在 MVP 阶段可恒为字符串描述,上线后再用业务错误码枚举)。前端用 `code` 判断跳登录 / 付费,**不要**用 `message` 文案。
3. **日志带可选 `user_id`**:MVP 阶段每条请求日志有一个恒为 `null` 的 `user_id` 字段,上线后填真实值。**不要**等到 M5 才改日志 schema。
4. **资源数据继续走只读文件系统**:词表 / 模板继续随代码部署,不上"用户级资源",避免后续重写。
5. **`Authorization` 头可选**:`/api/confuse` 在 MVP 阶段不读 `Authorization`,但后端解析时**不要拒绝**带头的请求(留作兼容)。
6. **`Cache-Control: no-store`**:在 MVP 阶段就要加,不能等上线后再说。

### 15.5 上线前必须敲定的待定项(沿用 §13.23~26)

| 编号 | 待定项 | 默认建议 |
|---|---|---|
| §13.23 | 计费维度 | 按"次"计费,简单;字符数方案留作扩展 |
| §13.24 | 支付厂商 | 国内:微信 + 支付宝;海外:Stripe(若不上线海外则不接) |
| §13.25 | 免费档配额 | 匿名:5 次/天;已注册:20 次/天;会员:无限 |
| §13.26 | 会员档权益 | 无限次 + `count > 10` 高批量 + 高级预设(所有 OJ 全开) + API token |

### 15.6 错误响应业务错误码(上线时定义,MVP 阶段为占位字符串)

```
parse_error            // 源码语法错
verify_error           // 编译/语法验证失败
transform_error        // 内部变换崩溃
quota_exceeded         // 配额用尽
auth_required          // 需要登录(匿名访问付费功能)
auth_invalid           // token 无效 / 过期
subscription_expired   // 会员过期
payment_required       // 触发付费墙
rate_limited           // 单 IP 频率超限
payload_too_large      // 超过 200KB
internal_error         // 兜底
```

### 15.7 合规清单(上线时必须完成)

- 用户协议(注册时勾选)。
- 隐私政策(明确"代码不持久化,只存元数据")。
- 注销账户功能(GDPR / 个保法"被遗忘权")。
- 付费相关的发票 / 退款政策。
- 未成年人保护(若允许注册,需年龄确认)。
- 数据导出(用户可导出自己的用量历史)。

### 15.8 API 开放(可选,上线后期)

MVP / M5 阶段不开放 API;若上线后考虑,需要:
- 单独的 API key(与登录态分离)。
- 按调用计费(更严格的配额)。
- 单独的速率限制档位。
